# CSV-импорт товаров

Гайд по загрузке товаров через страницу `/admin/import` и эндпоинт
`POST /api/admin/import/csv`. Все поведения, описанные здесь, соответствуют
текущему коду в `src/lib/server/services/csvImportService.js` и
`src/lib/server/repositories/productRepository.js`.

## 1. Как это работает в двух словах

1. CSV парсится **в браузере** (см. `src/lib/client/csv-parse.ts`) — поддерживаются
   разделители `,` и `;`, экранирование кавычками и BOM.
2. Распарсенные строки уходят на бэкенд как JSON: `{ rows: [...] }`.
3. Сервер по каждой строке вызывает `productRepository.upsertProductBySku(...)`:
   - **товар** ищется по `sku` — если есть, обновляется; если нет, создаётся;
   - **варианты** добавляются по правилу: `variant_sku = base_sku + хеш от атрибутов варианта`,
     `ON CONFLICT(sku) DO UPDATE` в `product_variants`.
4. Пустые ячейки **не перезатирают** существующие поля. То есть второй раз
   слать ту же карточку только ради варианта — безопасно.

> Обязательная колонка одна: `sku`. Для **новой** карточки также обязательно
> указать `category` (иначе сервер вернёт `category is required for new product`).

## 1а. Экспорт CSV

Выгрузка доступна на `/admin/products` и кнопкой «Скачать текущий каталог» на
`/admin/import`.

**Эндпоинт:** `GET /api/admin/products/export` (требуется сессия админки).

**Параметры** (те же, что у таблицы товаров):

| Параметр | Значение |
|---|---|
| `mode` | `import` (по умолчанию) или `full` |
| `search` | Поиск по SKU / названию |
| `categoryId`, `subcategoryId` | Фильтр категории |
| `manufacturer` | Производитель |
| `hit` | `1` / `0` — только хиты / без хита |
| `onSale` | `1` / `0` — только акционные / без акции |
| `attr_<code>` | Фильтр по характеристике |
| `ids` | Список id через запятую (выбранные товары) |

**Режим `import`** — формат совместим с импортом: одна строка на вариант,
повтор `sku`, пустые базовые поля на дополнительных строках. Можно выгрузить →
отредактировать в Excel → залить обратно через `/admin/import`.

**Режим `full`** — дополнительно: `id`, `slug`, `is_active`, `is_on_sale`,
`compare_at_price`, `badges`, `display_order`. Импорт эти колонки **пока не
обрабатывает** — режим для бэкапа и аналитики.

Файл отдаётся с UTF-8 BOM и разделителем `;` (корректно открывается в Excel RU).

## 2. Колонки CSV

| Колонка | Назначение |
|---|---|
| `sku` | Уникальный SKU базового товара. Повторите его в нескольких строках, чтобы добавить варианты. |
| `name` | Название карточки. |
| `category` | Категория. Поддерживает форму `Корень>>>Подкатегория` (см. §3). |
| `subcategory` | Подкатегория. Можно не заполнять, если она уже задана в `category`. |
| `price` | Цена базового товара (целое число). |
| `imageUrl` | Одно или несколько URL (через пробел). При `STORE_IMAGES_LOCALLY=1` картинки скачиваются в `public/uploads`. |
| `model_key` | Ключ модели: **одинаковый** `name` + **одинаковый** `model_key` связывают карточки для переключателей **цвета** и **стекла** на сайте (см. §4.2–4.3, §5). |
| `attr:<code>` | Характеристика **товара** (product-scope): `attr:color`, `attr:glass`, `attr:thickness`, `attr:manufacturer`. |
| `attr_name:<Название>` | То же самое, но по имени. Если атрибута нет — создастся **текстовым**. |
| `variant_attr:<code>` | **Ось варианта** (variant-scope): `variant_attr:size`, `variant_attr:opening`. Сохраняется в `product_variants.attrs`. |
| `variantSku` | Переопределяет вычисленный SKU варианта. |
| `variantPrice` | Переопределяет цену варианта. |
| `variantImageUrl` | Переопределяет картинку варианта. |
| `variantAttributes` | Альтернатива `variant_attr:*`: формат `code:value;code:value`. |

### Что под капотом «продуктовый» vs «вариантный» атрибут

В `attribute_definitions` каждый атрибут имеет колонку `scope`:
- `scope = 'product'` → значение хранится в `products.attrs` (JSONB).
- `scope = 'variant'` → значение хранится в `product_variants.attrs` (JSONB).

Сервис не доверяет префиксу слепо: если атрибут `code = size` помечен
в БД как `scope = variant`, то даже из `attr:size` он будет автоматически
маршрутизирован в вариант. Но **прямой и предсказуемый путь — использовать
`variant_attr:*` именно для осей вариантов**.

## 3. Категории и подкатегории

Три равнозначные формы:

```text
category=Входные двери>>>Премиум
```

```text
category=Входные двери
subcategory=Премиум
```

```text
category=Входные двери;Премиум
```

Любая из них даст один и тот же результат: товар попадёт в `category_id`,
указывающий на подкатегорию **Премиум** (которая, в свою очередь,
`parent_id` ссылается на **Входные двери**).

Если подкатегория не найдена — товар привяжется к корневой категории.
Несуществующая корневая категория **не создаётся автоматически** —
строка получит ошибку `категория не найдена`.

### Сидированный набор категорий

| Корень | Подкатегории |
|---|---|
| Входные двери | Премиум, Стандарт |
| Межкомнатные двери | Classic, Modern |
| Фурнитура | Ручки |

Новые категории и подкатегории создаются через админку (раздел на главной странице
`/admin`) **до** импорта.

## 4. Варианты: одна карточка — несколько строк

Чтобы у одного товара появилось несколько вариантов, **повторите ту же
`sku`** в нескольких строках и в каждой укажите свою комбинацию
`variant_attr:*`.

Базовые поля (`name`, `category`, `price`, `imageUrl`) достаточно
заполнить в **первой строке**. В последующих можно оставить пустыми —
они будут проигнорированы (см. п.1.4).

### 4.1. Входные двери: размер + открывание

Ось вариантов — `size` (option) и `opening` (option: «Левое»/«Правое»).
Один товар «Браво Сталь 01» с тремя вариантами:

```csv
sku,name,category,price,imageUrl,attr:color,attr:thickness,attr:manufacturer,variant_attr:size,variant_attr:opening
ENTRY-BRAVO-01,Браво Сталь 01,Входные двери>>>Премиум,79900,https://example.com/entry.jpg,Чёрный,80,Браво,860x2050,Левое
ENTRY-BRAVO-01,,,,,,,,860x2050,Правое
ENTRY-BRAVO-01,,,,,,,,960x2050,Левое
```

Что получится в БД:
- `products`: одна строка `ENTRY-BRAVO-01` с `attrs = { color, thickness, manufacturer }`.
- `product_variants`: три строки с SKU вида `ENTRY-BRAVO-01--<hash>` и
  `attrs = { size, opening }`.

На карточке появятся два селектора — «Размер» и «Открывание».

### 4.2. Межкомнатные двери: размер + цвет + стекло (`model_key`)

С межкомнатными **цвет** и **стекло** — атрибуты **товара** (`product-scope`):
каждое сочетание «цвет × стекло» — отдельная строка в `products` со своим `sku`.
**Размер** (и при необходимости открывание) остаётся в **вариантах** (`variant_attr:size`).

Идиома как для цвета: у всех карточек одной модели совпадают **`name`** и **`model_key`**.
Тогда:

- на карточке товара API отдаёт **`colorVariants`** — соседи с **тем же стеклом** (если
  `attr:glass` заполнен), чтобы в переключателе цвета не смешивались разные вставки;
- и **`glassVariants`** — соседи с **тем же цветом**, если у модели несколько значений
  `attr:glass`;
- в **каталоге** (выдаче) у карточки показываются **чипы стёкол** (`glassOptions`), если
  для этой модели и цвета в базе больше одного непустого значения стекла.

```csv
sku,name,category,price,imageUrl,model_key,attr:color,attr:glass,variant_attr:size
INT-M-01-W-CLR,Classic 01,Межкомнатные двери>>>Classic,18900,https://example.com/a.jpg,classic-m-01,Белый,Прозрачное,800x2000
INT-M-01-W-CLR,,,,,,,,900x2000
INT-M-01-W-MAT,Classic 01,Межкомнатные двери>>>Classic,19200,https://example.com/b.jpg,classic-m-01,Белый,Матовое,800x2000
INT-M-01-W-MAT,,,,,,,,900x2000
INT-M-01-O-CLR,Classic 01,Межкомнатные двери>>>Classic,19500,https://example.com/c.jpg,classic-m-01,Орех,Прозрачное,800x2000
```

Что получится:

- три карточки в `products` с одним `name` и `model_key`, различаются по `color` / `glass`;
- у каждой карточки — свои строки в `product_variants` (размеры);
- на сайте: переключатель **цвета** (при заполненном стекле — только цвета «при этом стекле»),
  переключатель **стекла** (только стёкла «при этом цвете»), селектор **варианта** по размеру.

Рекомендации по данным:

- Заведите атрибут **`glass`** (или другой код, согласованный в `attribute_definitions`)
  как `scope = product`, тип `option` или `text`, и заполняйте **`attr:glass`** в CSV.
- Для **каждого** варианта стекла задавайте явное значение (в т.ч. вариант вроде
  «Без вставки»), чтобы переключатели и фильтры в каталоге вели себя предсказуемо.

> Группировка переключателей работает только при **совпадении `name` и `model_key`**.
> Если `model_key` пустой — карточки не связываются.

### 4.3. Альтернатива через `variantAttributes`

Если генератор CSV не хочет плодить колонки `variant_attr:*`, можно
сложить всё в одну ячейку:

```csv
sku,name,category,price,variantAttributes
ENTRY-BRAVO-01,Браво Сталь 01,Входные двери>>>Премиум,79900,size:860x2050;opening:Левое
ENTRY-BRAVO-01,,,,size:860x2050;opening:Правое
```

Разделитель пар — `;`, разделитель ключ/значение — `:`.

## 5. Группировка модели (`model_key`): цвет и стекло

Колонка `model_key` пишется в `products.model_key`. При открытии карточки
`getProductById(id)` подтягивает соседей с тем же **`model_key` и `name`**:

1. **`colorVariants`** — другие карточки той же модели; если у текущей карточки
   **заполнен** `attrs.glass`, в списке остаются только соседи с **тем же стеклом**;
   если стекло **не задано** — только соседи **без стекла** (пустой `glass`), чтобы
   не смешивать линейку «без вставки» с карточками со стеклом.
2. **`glassVariants`** — соседи с **тем же цветом** (если цвет задан — только он;
   если цвет пуст — только соседи без цвета); среди них **более одного** непустого
   `attrs.glass` — иначе блок на карточке не показывается.

В **списке каталога** (`GET /api/products`) для каждой позиции считается
`glassOptions`: пары `{ id, label }` по соседям с тем же `model_key`, `name` и
`color` — под чипами «Стекло» на плитке витрины.

Правила:

- Пустая ячейка `model_key` ничего не перезатирает (как и любая другая колонка).
- Цвет по-прежнему кладите в `attr:color`, стекло — в **`attr:glass`** (код в БД
  должен совпадать с тем, что ожидает фронт; в сиде и примерах используется `glass`).
- Для межкомнатных с переключателями у всех карточек одной модели должны совпадать
  **`name` и `model_key`**.

## 5а. Комплектующие / погонаж (`pogonazh_id`)

Межкомнатные двери продаются вместе с **наличниками, коробками, доборами**.
Привязка «полотно ↔ погонаж» делается через текстовый атрибут
**`pogonazh_id`** (название в админке — «Идентификатор погонажа»).

Идея:

- У каждого **полотна** заполняем `attr:pogonazh_id` — код группы аксессуаров
  (например, `824` для Stormy Silk).
- У каждого **комплектующего** (карточка лежит в категории «Погонаж»)
  заполняем тот же `attr:pogonazh_id` и нужные атрибуты — `attr:color`,
  длина / ширина / толщина и т.п. (см. свою маппинг-таблицу при загрузке).
- На карточке полотна `GET /api/products/:id` возвращает массив
  **`accessories`** — все активные товары с **тем же значением** `pogonazh_id`,
  кроме самого полотна. Под товаром появляется блок «Комплектующие в этом цвете»
  с количеством и кнопкой «Добавить в корзину».

> У одного SKU может быть **несколько значений `pogonazh_id`** (например,
> одна и та же коробка подходит для трёх цветов). Хранятся они в одной ячейке
> JSONB как строка, разделённая запятыми/точкой с запятой/пробелом, например
> `824,826,871`. Совпадение для блока «Комплектующие» считается **как пересечение
> множеств**: достаточно одного общего ID, чтобы товар попал в аксессуары.

### Как добавляются значения при импорте

Правило простое: **при CSV-апсерте `pogonazh_id` не перезаписывается, а
дополняется**. Это удобно, когда у одного SKU несколько групп аксессуаров:

- Если в **одной** заливке CSV один и тот же SKU встречается несколько раз с
  разными `attr:pogonazh_id`, итог в БД — объединение всех непустых ID без
  дубликатов.
- Если SKU уже есть в БД с каким-то `pogonazh_id`, а в новой заливке вы укажете
  другой — он **добавится** к существующему, а не заменит его. Повторный ID
  будет проигнорирован (дубликаты схлопываются).
- Пустая ячейка `attr:pogonazh_id` (как и любая другая) ничего не перезаписывает.
- Очистить или переписать список можно вручную: через `/admin/products` → правка
  карточки → атрибут «Идентификатор погонажа» (там лежит сырая строка `824,826`).

Пример: коробка `170-0008`, подходящая для трёх цветов, может быть залита одной
заливкой с тремя строками одного и того же SKU:

```csv
sku,name,category,price,attr:color,attr:pogonazh_id
170-0008,Коробка телескоп 2070*70*30,Погонаж,1190,Stormy Silk,824
170-0008,Коробка телескоп 2070*70*30,Погонаж,1190,Stormy Silk,826
170-0008,Коробка телескоп 2070*70*30,Погонаж,1190,Stormy Silk,871
```

После импорта в БД у SKU `170-0008` будет `attrs.pogonazh_id = "824,826,871"`.

Пример CSV (фрагмент): полотно + три комплектующих в Stormy Silk (`pogonazh_id = 824`):

```csv
sku,name,category,price,imageUrl,model_key,attr:color,attr:pogonazh_id,variant_attr:size
INT-BRAVO-0-STORMY,Браво-0,Межкомнатные двери>>>Экошпон,4920,https://example.com/d.jpg,bravo-0,Stormy Silk,824,200x60
170-0007,Добор «Т» 2070*100*8,Погонаж,795,,,Stormy Silk,824,
170-0008,Коробка телескоп 2070*70*30,Погонаж,1190,,,Stormy Silk,824,
170-0009,Наличник телескоп 2200*70*8,Погонаж,640,,,Stormy Silk,824,
```

Заметки:

- Колонка «Идентификатор» (внутренний код 1С) — **не** маппится в БД,
  её при импорте отмечайте как «пропустить колонку» (см. скриншот мастера).
- Колонки «Идентификатор группы аксессуров» и «Название группы аксессуров»
  идут соответственно в `attr:pogonazh_id` и `attr:color`.
- Атрибут `pogonazh_id` нужен **в `attribute_definitions` заранее** — он есть
  в сиде (`initSchema.js`); если его в БД нет, создайте через `/admin`
  или включите колонку как `attr_name:Идентификатор погонажа` — тогда импорт
  создаст текстовый атрибут на лету.

## 6. Чек-лист перед запуском

- [ ] Все нужные категории и подкатегории заведены в админке.
- [ ] Атрибуты (`color`, `glass`, `thickness`, `size`, `opening`, …) существуют и имеют
      правильный `scope`. Несуществующий код в `attr:<code>` — строка получает
      ошибку; `attr_name:<Имя>` создаст текстовый атрибут на лету.
- [ ] Цены — целые числа (минор-единицы или рубли — на ваше усмотрение,
      главное единообразно).
- [ ] У строк-«добавок варианта» **повторяется тот же `sku`** и не заполнены
      базовые поля, чтобы случайно не перезаписать карточку.
- [ ] Для межкомнатных с группировкой по модели у всех связанных карточек
      **одинаковые `name` и `model_key`**; при нескольких стёклах заполнен **`attr:glass`**.

## 7. Что делает страница `/admin/import`

1. Принимает CSV (файл или вставленный текст), парсит на клиенте.
2. Показывает превью первых 10 строк и все распознанные колонки.
3. По кнопке «Импортировать» отправляет `POST /api/admin/import/csv`.
4. Ответ сервера:
   ```json
   { "ok": true, "imported": 7, "errors": [] }
   ```
   Если в `errors` что-то есть — конкретный номер строки и причина
   будут выведены на странице.

## 8. Минимальный валидный пример

```csv
sku,name,category,price
DEMO-1,Тестовая дверь,Входные двери>>>Премиум,10000
```

Этого достаточно, чтобы создать карточку: атрибутов и вариантов у неё пока нет,
их можно дозалить повторным импортом с тем же `sku`.
