# CSV-импорт товаров

Гайд по загрузке товаров через страницу `/admin/import` и эндпоинт
`POST /api/admin/import/csv`. Все поведения, описанные здесь, соответствуют
текущему коду в `src/lib/server/services/csvImportService.js` и
`src/lib/server/repositories/productRepository.js`.

## 1. Как это работает в двух словах

1. CSV парсится **в браузере** (см. `src/lib/client/csv-parse.ts`) — поддерживаются
   разделители `,` и `;`, экранирование кавычками и BOM.
2. Распарсенные строки уходят на бэкенд как JSON: `{ rows: [...] }`.
3. Сервер по каждой строке вызывает `productRepository.upsertProductBySku(...)`:
   - **товар** ищется по `sku` — если есть, обновляется; если нет, создаётся;
   - **варианты** добавляются по правилу: `variant_sku = base_sku + хеш от атрибутов варианта`,
     `ON CONFLICT(sku) DO UPDATE` в `product_variants`.
4. Пустые ячейки **не перезатирают** существующие поля. То есть второй раз
   слать ту же карточку только ради варианта — безопасно.

> Обязательная колонка одна: `sku`. Для **новой** карточки также обязательно
> указать `category` (иначе сервер вернёт `category is required for new product`).

## 2. Колонки CSV

| Колонка | Назначение |
|---|---|
| `sku` | Уникальный SKU базового товара. Повторите его в нескольких строках, чтобы добавить варианты. |
| `name` | Название карточки. |
| `category` | Категория. Поддерживает форму `Корень>>>Подкатегория` (см. §3). |
| `subcategory` | Подкатегория. Можно не заполнять, если она уже задана в `category`. |
| `price` | Цена базового товара (целое число). |
| `imageUrl` | Одно или несколько URL (через пробел). При `STORE_IMAGES_LOCALLY=1` картинки скачиваются в `public/uploads`. |
| `model_key` | Ключ модели для **группировки цветовых вариантов** в одну карточку (см. §5). |
| `attr:<code>` | Характеристика **товара** (product-scope): `attr:color`, `attr:thickness`, `attr:manufacturer`. |
| `attr_name:<Название>` | То же самое, но по имени. Если атрибута нет — создастся **текстовым**. |
| `variant_attr:<code>` | **Ось варианта** (variant-scope): `variant_attr:size`, `variant_attr:opening`. Сохраняется в `product_variants.attrs`. |
| `variantSku` | Переопределяет вычисленный SKU варианта. |
| `variantPrice` | Переопределяет цену варианта. |
| `variantImageUrl` | Переопределяет картинку варианта. |
| `variantAttributes` | Альтернатива `variant_attr:*`: формат `code:value;code:value`. |

### Что под капотом «продуктовый» vs «вариантный» атрибут

В `attribute_definitions` каждый атрибут имеет колонку `scope`:
- `scope = 'product'` → значение хранится в `products.attrs` (JSONB).
- `scope = 'variant'` → значение хранится в `product_variants.attrs` (JSONB).

Сервис не доверяет префиксу слепо: если атрибут `code = size` помечен
в БД как `scope = variant`, то даже из `attr:size` он будет автоматически
маршрутизирован в вариант. Но **прямой и предсказуемый путь — использовать
`variant_attr:*` именно для осей вариантов**.

## 3. Категории и подкатегории

Три равнозначные формы:

```text
category=Входные двери>>>Премиум
```

```text
category=Входные двери
subcategory=Премиум
```

```text
category=Входные двери;Премиум
```

Любая из них даст один и тот же результат: товар попадёт в `category_id`,
указывающий на подкатегорию **Премиум** (которая, в свою очередь,
`parent_id` ссылается на **Входные двери**).

Если подкатегория не найдена — товар привяжется к корневой категории.
Несуществующая корневая категория **не создаётся автоматически** —
строка получит ошибку `категория не найдена`.

### Сидированный набор категорий

| Корень | Подкатегории |
|---|---|
| Входные двери | Премиум, Стандарт |
| Межкомнатные двери | Classic, Modern |
| Фурнитура | Ручки |

Новые категории и подкатегории создаются через админку
(`/admin/categories`, `/admin/categories/[id]/subcategories`)
**до** импорта.

## 4. Варианты: одна карточка — несколько строк

Чтобы у одного товара появилось несколько вариантов, **повторите ту же
`sku`** в нескольких строках и в каждой укажите свою комбинацию
`variant_attr:*`.

Базовые поля (`name`, `category`, `price`, `imageUrl`) достаточно
заполнить в **первой строке**. В последующих можно оставить пустыми —
они будут проигнорированы (см. п.1.4).

### 4.1. Входные двери: размер + открывание

Ось вариантов — `size` (option) и `opening` (option: «Левое»/«Правое»).
Один товар «Браво Сталь 01» с тремя вариантами:

```csv
sku,name,category,price,imageUrl,attr:color,attr:thickness,attr:manufacturer,variant_attr:size,variant_attr:opening
ENTRY-BRAVO-01,Браво Сталь 01,Входные двери>>>Премиум,79900,https://example.com/entry.jpg,Чёрный,80,Браво,860x2050,Левое
ENTRY-BRAVO-01,,,,,,,,860x2050,Правое
ENTRY-BRAVO-01,,,,,,,,960x2050,Левое
```

Что получится в БД:
- `products`: одна строка `ENTRY-BRAVO-01` с `attrs = { color, thickness, manufacturer }`.
- `product_variants`: три строки с SKU вида `ENTRY-BRAVO-01--<hash>` и
  `attrs = { size, opening }`.

На карточке появятся два селектора — «Размер» и «Открывание».

### 4.2. Межкомнатные двери: размер + цвет (две карточки + `model_key`)

С межкомнатными немного хитрее, потому что **цвет — это product-scope атрибут**
(каждый цвет фактически меняет SKU модели), а **размер — variant-scope**.

Идиома: завести **по карточке на цвет** и связать их одним `model_key` +
одинаковым `name` — тогда на фронте появится переключатель цвета между
карточками одной модели.

```csv
sku,name,category,price,imageUrl,model_key,attr:color,variant_attr:size
INT-CLASSIC-WHITE,Classic 01,Межкомнатные двери>>>Classic,18900,https://example.com/cl-w.jpg,classic-01,Белый,800x2000
INT-CLASSIC-WHITE,,,,,classic-01,,900x2000
INT-CLASSIC-OAK,Classic 01,Межкомнатные двери>>>Classic,19500,https://example.com/cl-o.jpg,classic-01,Дуб,800x2000
INT-CLASSIC-OAK,,,,,classic-01,,900x2000
```

Что получится:
- Две карточки в `products` — `INT-CLASSIC-WHITE` (Белый) и `INT-CLASSIC-OAK` (Дуб)
  с одинаковым `name = "Classic 01"` и `model_key = "classic-01"`.
- У каждой — по два варианта в `product_variants` (размеры 800x2000 / 900x2000).
- В выдаче `getProductById` фронт получит блок `colorVariants` — пользователь
  сможет переключаться между цветами **одной модели**.

> Группировка цветов работает только если у карточек **одновременно
> совпадают `name` И `model_key`** (см. SQL в `productRepository.getProductById`).
> Если оставить `model_key` пустым — каждая карточка будет жить отдельно.

### 4.3. Альтернатива через `variantAttributes`

Если генератор CSV не хочет плодить колонки `variant_attr:*`, можно
сложить всё в одну ячейку:

```csv
sku,name,category,price,variantAttributes
ENTRY-BRAVO-01,Браво Сталь 01,Входные двери>>>Премиум,79900,size:860x2050;opening:Левое
ENTRY-BRAVO-01,,,,size:860x2050;opening:Правое
```

Разделитель пар — `;`, разделитель ключ/значение — `:`.

## 5. Группировка цветовых вариантов (`model_key`)

Колонка `model_key` пишется в `products.model_key` и используется только
для одного: при открытии карточки `getProductById(id)` дополнительно
подтягивает все товары, у которых **`model_key` совпадает И `name` совпадает**,
и формирует блок `colorVariants`.

Правила:
- Пустая ячейка `model_key` ничего не перезатирает (как и любая другая колонка).
- Чтобы **снять** ключ модели у уже импортированного товара, поставьте в ячейку
  один пробел/тире — нет, проще через админку: импорт пустую строку трактует
  как «оставь как было».
- Цвет товара всё равно нужно положить в `attr:color`, иначе переключателю
  нечего показывать.

## 6. Чек-лист перед запуском

- [ ] Все нужные категории и подкатегории заведены в админке.
- [ ] Атрибуты (`color`, `thickness`, `size`, `opening`, …) существуют и имеют
      правильный `scope`. Несуществующий код в `attr:<code>` — строка получает
      ошибку; `attr_name:<Имя>` создаст текстовый атрибут на лету.
- [ ] Цены — целые числа (минор-единицы или рубли — на ваше усмотрение,
      главное единообразно).
- [ ] У строк-«добавок варианта» **повторяется тот же `sku`** и не заполнены
      базовые поля, чтобы случайно не перезаписать карточку.
- [ ] Для межкомнатных, где нужна группировка цветов, у всех цветовых карточек
      одной модели **одинаковые `name` и `model_key`**.

## 7. Что делает страница `/admin/import`

1. Принимает CSV (файл или вставленный текст), парсит на клиенте.
2. Показывает превью первых 10 строк и все распознанные колонки.
3. По кнопке «Импортировать» отправляет `POST /api/admin/import/csv`.
4. Ответ сервера:
   ```json
   { "ok": true, "imported": 7, "errors": [] }
   ```
   Если в `errors` что-то есть — конкретный номер строки и причина
   будут выведены на странице.

## 8. Минимальный валидный пример

```csv
sku,name,category,price
DEMO-1,Тестовая дверь,Входные двери>>>Премиум,10000
```

Этого достаточно, чтобы создать карточку: атрибутов и вариантов у неё пока нет,
их можно дозалить повторным импортом с тем же `sku`.
